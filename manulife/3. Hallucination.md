🔐 5. Security / Hallucination (critical)

Modern LLMs are primarily based on Transformer architectures, Attention is All you Need. While powerful, this design inherently introduces the risk of hallucination. As such, it cannot be fully resolved within the model itself, and prompt engineering alone is insufficient for production-grade reliability.

While, blaming the model alone is not fair, since LLM is only one component of our system. Our system as a whole suppose to provide a broader and controlled pipeline to do the job right. LLM itself alone contribute its part.

Based on practical experience and iteration, I’ve adopted the following strategies:

My strategies include:

1. Input Control (Garbage In, Garbage Out)
I enforce strict input normalization and context curation before inference. This typically involves preprocessing pipelines and RAG to anchor the model’s outputs in verified, domain-relevant data. The goal is to reduce entropy in the input space and constrain generation. 

2. Inspired by RLHF, I strongly support Human-in-the-Loop Feedback
I integrate human evaluation into the loop by capturing outputs, ranking them, and storing validated corrections. Over time, this creates a deterministic fallback layer that overrides model variability in recurring scenarios. 

For instance, in a COA classification task, the model may assign different accounts to the same vendor (e.g., Costco) depending on context, which may be acceptable in enterprise settings but not for SMEs. To address this, I added a guardrail that enforces deterministic mappings (e.g., Costco → Office Supplier) before persistence, ensuring alignment with domain-specific requirements.
Overall, the key principle is shifting from model-centric trust to system-enforced reliability, combining probabilistic generation with deterministic controls.

Providng the user traceability — providing  verifiable and auditable evidence, confidence, and citation. basically training user to cope with AI model. 

📊 Evaluation Dimensions
Faithfulness (0.8): Measures how well the generated answer aligns with the retrieved context
Answer Relevancy (0.9): Evaluates whether the response directly addresses the user query
Context Precision (0.7): Assesses how much of the retrieved context is actually relevant
Context Recall (0.85): Measures whether all necessary information was successfully retrieved

def has_hallucination_keywords(answer, context):
    """
    检测回答中是否存在原文没有的关键词
    """
    answer_keywords = extract_keywords(answer)
    context_keywords = extract_keywords(context)
    
    # 回答关键词是否都在上下文中
    return not answer_keywords.issubset(context_keywords)

自我一致性检查：
def self_consistency_check(question, context, answer, n=3):
    """
    让模型多次生成答案，检查一致性
    """
    answers = []
    for _ in range(n):
        ans = llm.generate(question, context)
        answers.append(ans)
    
    # 如果多次回答差异很大 → 可能存在幻觉
    return calculate_similarity(answers)

六、RAG 幻觉 vs 大模型原生幻觉
对比项	RAG 幻觉	原生幻觉
信息来源	检索文档 + 模型内部知识	仅模型内部知识
可控性	相对较高（可优化检索）	较低
检测难度	可与原文对比，相对容易	需外部知识库验证
缓解方法	改进检索、强制引用	RLHF、Prompt 工程
典型错误	数字扭曲、引用错位	事实编造、逻辑错误

八、一句话总结
RAG 的幻觉本质是 "生成模型不听话" —— 即使给了参考答案，它还是会自由发挥。缓解的核心是：强制约束 + 多重验证。

最佳实践公式：

高质量 RAG = 精准检索 × 强约束 Prompt × 生成后校验
